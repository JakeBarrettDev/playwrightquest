## What this project is

**PlaywrightStagecraft** is a browser-based game that teaches people how to write
Playwright tests against fictitious websites. Players write real Playwright test code
in a Monaco editor, run it against a real Chromium browser (via Docker), and receive
AI-powered grading and tutoring feedback. It uses a BYOK (Bring Your Own Key) model —
players supply their own Claude/OpenAI/Gemini API key for the AI judge.

The number one product priority is that **the AI tutor gives real, specific, impactful
feedback grounded in actual Playwright best practices**. Everything else is secondary.

The project is a Next.js 16 app (App Router) with TypeScript, Tailwind CSS 4, and Monaco
editor. It lives at: `/Users/jakebarrett/Desktop/projects/playwrightStagecraft/playwrightquest`

The owner is Jake (maroonraccoondev@gmail.com). His collaborator Liz may also continue work.

## Key architectural decisions made this session

**Deployment target: Railway at $5/month.** Vercel is a hard no — no Docker support,
ephemeral filesystem breaks trace persistence, function timeout too short for Playwright.
Railway runs a persistent container, supports the full Docker execution model, deploys
from GitHub.

**Production execution model: LocalRunner inside a Playwright container.** Rather than
Docker-in-Docker (the app spawning `docker run` from inside a container), the Dockerfile
uses `mcr.microsoft.com/playwright:v1.59.1-noble` as its base image. The app IS the
Playwright environment. `EXECUTION_RUNNER=local` in production. No Docker socket access
needed, no container startup latency per run, simpler and faster.

**Trace viewer: video-based, not screenshot-based.** Playwright's per-action snapshots
are HTML DOM snapshots, not PNGs. The trace player uses `video: 'on'` and scrubs a
`<video>` element to each action's timestamp. Bounding boxes from the trace overlay the
video at the correct scale.

**Donation model for hosting costs.** The app is free. A donation page (GitHub Sponsors
or Ko-fi) covers the ~$5/month Railway cost. Performance is not compromised to save money.

**BYOK stays.** AI grading/tutoring costs are on the user. The server only pays for
compute (test execution), not per-grading AI calls.

## Things to know before continuing any work

### The corpus is the heart of the grader

Every piece of feedback the AI gives is grounded in corpus files. If grading quality
seems off, look at `corpus/` first. The `grader-instructions.md` is the system prompt —
edit it to change the AI's stance. The `rubric-rules/` files control scoring. The
`challenge-notes/` files document intentional traps per site.

### The trace format is Playwright v1.59 specific

The parser in `lib/trace/parser.ts` reads `trace.trace` (JSON-LD events) from the
trace.zip. It pairs `before`/`input`/`after` events by `callId`. The `sourceLine` comes
from `frames[]` — it finds the frame whose `file` ends with `player.spec.ts`. Bounding
boxes come from `input.targetBoundingBox`. Video timestamps are computed relative to the
first action's `startTime`.

### The runId format

`randomUUID().replace(/-/g, "")` — 32 lowercase hex chars. The route validation regex
`/^[a-f0-9]{32}$/` matches this exactly.

### Multi-browser is wired but not exposed in the UI

The `browser` field on `ExecutionRequest` flows through to the Docker config's
`projects[]` array. The UI doesn't currently expose browser selection. Firefox and
WebKit binaries are in the Playwright Docker image and will work when the UI is added.

### The grading pipeline is untouchable

The flow is: `POST /api/grade` → `assembleGradingContext` → `renderContextBlocks` →
provider adapter → `GradingResult`. Do not modify this pipeline lightly. The prefix
caching strategy (authoritative/site/dynamic blocks) is intentional and reduces token
costs significantly.

### Local development

```bash
npm install
npm run dev
# For Docker execution:
EXECUTION_RUNNER=docker npm run dev
# Docker must be running and the Playwright image must be pulled:
docker pull mcr.microsoft.com/playwright:v1.59.1-noble
```
